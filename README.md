### （1）题目意思与生僻概念解释

**题目意思：**
给出一个未排序的整数数组，要求找出其中**没有出现过的、最小的正整数**（正整数从 1 开始：1, 2, 3...）。
例如：数组 `[3, 4, -1, 1]` 中，出现了 1、3、4，缺失的最小正整数就是 2。
并且题目通常会附加极其苛刻的限制条件：时间复杂度必须是 $O(n)$，且只能使用 $O(1)$ 的额外常数空间。

**核心生僻概念解释：**
* **原地哈希（In-place Hashing） / 抽屉原理：** 既然只能用 $O(1)$ 的空间，就不能额外开辟一个哈希表去记录哪些数字出现过。核心思想是**把数组自身当成哈希表（抽屉）**。规则是：数字 `1` 应该放在下标 `0` 的位置，数字 `2` 应该放在下标 `1` 的位置……即数字 `x` 应该放在下标 `x - 1` 的位置。
* **双指针区间划分：**
    * **左指针 `left`：** 维护“已归位的有效正数区”。区间 `[0, left)` 里的数字绝对是完美的 `1, 2, 3...left`。
    * **右指针 `right`：** 维护“垃圾区”。区间 `[right, n)` 里的数字都是经过判定，对寻找当前连续正数序列**毫无帮助**的废数字。
    * **中间区间 `[left, right)`：** 尚未被检查、等待处理的盲区。

### （2）代码解题思路解析

代码的核心在于不断审视当前指针 `left` 所指的数字 `nums[left]`，并根据它的价值将其分为三类进行处理，直到盲区消失（`left == right`）：

1.  **场景一（完美归位）：** 如果 `nums[left]` 恰好等于 `left + 1`，说明这个数字正好在它该在的位置上。有效区向右扩张（`left += 1`）。
2.  **场景二（判定为垃圾）：** 遇到以下三种情况，直接将其视为“垃圾”，将其与盲区最右侧的数字（下标为 `right - 1`）交换，并将垃圾区向左扩张（`right -= 1`）：
    * `nums[left] <= left`：非正数（≤ 0），或者是比当前 `left` 还要小的值。因为 `[0, left)` 已经排好了完美序列，再来一个更小的正数只可能是重复的无用数字。
    * `nums[left] > right`：数字大得离谱。当前盲区加有效区总共只剩下 `right` 个槽位，最多只能凑出从 1 到 `right` 的连续正数。一旦遇到大于 `right` 的数，它绝对插不进当前的完美序列中。
    * `nums[nums[left] - 1] == nums[left]`：要去的那个“抽屉”里，已经躺着一个和它一模一样的数字了（重复）。
3.  **场景三（有用但放错位置）：** 既不是当前位置的完美数字，也不是垃圾。说明它是一个有价值的数字，只是放错了地方。将其与它该去的位置（下标 `nums[left] - 1`）上的数字**交换**。注意，换完之后 `left` 指针**不移动**，因为从远方换过来的新数字还需要在下一轮继续被审视。

### （3）带有注释的代码框

```python
class Solution:
    def firstMissingPositive(self, nums: list[int]) -> int:
        # left 指针：有效区的右边界，[0, left) 内的元素保证是 1, 2, 3...
        left = 0
        # right 指针：垃圾区的左边界，[right, len(nums)) 内的元素被判定为无用。注意这里不需要 - 1。
        right = len(nums)  
        # 当盲区 [left, right) 还存在时，继续循环处理
        while left < right:
            # 场景 1: 当前位置的数字恰好等于期望值（下标 left 对应数字 left + 1）
            if nums[left] == left + 1:
                left += 1  # 有效区向右扩张
            # 场景 2: 当前数字是垃圾 -> 负数/过小、过大、目标位置已有重复数字
            elif nums[left] <= left or nums[left] > right or nums[nums[left] - 1] == nums[left]:
                right -= 1  # 垃圾区向左扩张
                # 将被判定为垃圾的当前数字，与盲区最右侧的未知数字交换
                nums[left], nums[right] = nums[right], nums[left]
            # 场景 3: 数字有效且未归位 -> 将其送到它该去的下标 (nums[left] - 1)
            else:
                # 提前提取目标索引，防止 Python 多重赋值特性带来的覆盖 bug
                target_index = nums[left] - 1
                # 执行交换操作，注意 left 指针停在原地，等待下一轮审视被换过来的新数字
                nums[left], nums[target_index] = nums[target_index], nums[left]
        # 循环结束，有效区 [0, left) 存放了 1 到 left 的完美序列
        # 因此，缺失的最小正数必然是紧随其后的 left + 1
        return left + 1
```

### 二、 用一个神仙算例看懂 `elif` 的三个条件
这行代码 `elif nums[left] <= left or nums[left] > right or nums[nums[left] - 1] == nums[left]:`
包含了三种被判定为“垃圾”的情况。设计一个数组：**`nums = [5, 0, 3, 3]`**。
长度是 4。初始 `left = 0`, `right = 4`。
在这个例子中，**`left` 一步都不会走，就在 `left = 0` 的位置，连续触发这三种垃圾判定！**

| 轮次 | 当前 `nums[0]` | 当前 `right` | 触发了哪个 `elif` 垃圾判定？ | 物理意义解释 | 执行动作与交换后的数组 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **1** | **5** | 4 | **`nums[left] > right`**<br>(5 > 4 成立) | **“数字过大”**。<br>数组一共就 4 个坑，最多排到 1,2,3,4。 5绝对是没法存的。 | `right -= 1` (变 3)<br>5 和 `nums[3]`(即 3) 交换。<br>👉 数组变为：`[3, 0, 3, 5]` |
| **2** | **3** | 3 | **`nums[nums[left]-1] == nums[left]`**<br>(即 `nums[2] == 3` 成立) | **“发现重复”**。<br>这个 3 看起来是个好数字，应该去下标 2 的位置。但是 `nums[2]`，发现那里已经躺着一个 3 了！重复的数字也不行。 | `right -= 1` (变 2)<br>3 和 `nums[2]`(即 3) 交换。<br>👉 数组变为：`[3, 0, 3, 5]` |
| **3** | **3** | 2 | **`nums[left] > right`**<br>(3 > 2 成立) | **“坑位缩水，好人变垃圾”**。<br>还是这个 3，刚才它只是重复，现在由于 `right` 缩减到了 2，说明盲区加有效区只剩 2 个位置了，最多只能排 1,2。此时的 3 变成了超大垃圾。 | `right -= 1` (变 1)<br>3 和 `nums[1]`(即 0) 交换。<br>👉 数组变为：`[0, 3, 3, 5]` |
| **4** | **0** | 1 | **`nums[left] <= left`**<br>(0 $\le$ 0 成立) | **“数字过小/非正数”**。<br>我们需要的是从 1 开始的正数，0 或者负数全都是垃圾。 | `right -= 1` (变 0)<br>0 和 `nums[0]` 和 0 交换。<br>👉 数组变为：`[0, 3, 3, 5]` |

**最终结果：**
经历了 4 次垃圾清理后，`right` 变成了 0。
此时 `left < right` (`0 < 0`) 不成立，循环直接结束。
返回 `left + 1`，也就是 `0 + 1 = 1`。
完美算出 `[5, 0, 3, 3]` 中缺失的第一个正数就是 **1**！

12. `else:`
    * **解释：** 排除上述所有情况，说明这个数字是个“好人”，应该去填补前面还没填上的某个坑位。
13. `target_index = nums[left] - 1`
    * **解释：** 计算这个好人该回的老家下标。在 Python 中，如果直接写 `nums[left], nums[nums[left]-1] = ...`，由于左到右的赋值计算顺序，`nums[left]` 先被改变后，后面的索引 `nums[left]-1` 也会跟着突变，导致位置错乱。必须单独提取。
14. `nums[left], nums[target_index] = nums[target_index], nums[left]`
    * **解释：** 将当前数字送回老家，同时把老家原本占位的另一个无辜数字换回 `left` 的位置。由于换回来的新数字还没被检查，所以下一轮 `while` 循环将继续在不动 `left` 的情况下审视它。

nums[left], nums[target_idx] = nums[target_idx], nums[left] 是：让流浪的数字回老家，同时把占据老家的陌生人揪出来。
用一个最简单的、只有 3 个数字的数组来演示这个动作。
** 设定算例：**
数组 `nums = [3, 1, 4]`
当前指针 `left = 0`。

1. 找目标
* **当前手里抓着谁？** `nums[left]` 就是 `nums[0]`，也就是数字 **`3`**。
* 数字 `3` 应该去下标 `2`。
* 所以，计算出它的目标下标：`target_idx = nums[left] - 1 = 2`。

2. 准备交换前，看看老家住着谁？
在动手换之前，先看看下标 `2` 的位置上是谁：是数字 **`4`**。

| 下标索引 | 0 (即 `left`) | 1 | 2 (即 `target_idx`) |
| :--- | :--- | :--- | :--- |
| **里面的数字** | **3** (想去下标 2) | 1 | **4** (目前占着下标 2) |

3. 执行交换魔法
代码执行：`nums[0], nums[2] = nums[2], nums[0]`
这个动作是**瞬间同时完成**的：
* 数字 **`3`** 飞到了下标 `2`。
* 原本在下标 `2` 的数字 **`4`**，被拎回了下标 `0`（交给了 `left` 指针）。

4. 交换后的结果与下一步的伏笔

| 下标索引 | 0 (即 `left`) | 1 | 2 (即 `target_idx`) |
| :--- | :--- | :--- | :--- |
| **里面的数字** | **4** (新来的，待审查) | 1 | **3** (完美归位！) |

**核心精髓来了：为什么换完之后，`left` 指针绝对不能加 1？**

看此时的数组变成了 `[4, 1, 3]`。
数字 `3` 确实回到了正确的坑位，**但是！** 刚刚被拎到 `nums[0]` 位置的那个数字 **`4`**，是一个彻头彻尾的“陌生人”。还没检查过它是不是垃圾，也不知道它该去哪。

所以，在这行代码执行完毕后，当前轮次结束。下一轮 `while` 循环开始时，`left` 依然停留在 `0`，算法会继续用同样的眼光去审视新换过来的数字 **`4`**（发现它超出了长度 3 的限制，判定为过大垃圾，扔到后面去）。
